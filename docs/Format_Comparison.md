# データフォーマット比較: ETON vs TOON vs JSON

LLM 通信における各フォーマットの特性を比較します。

## 1. 概要比較 (Comparison Overview)

| 特徴 | JSON (Standard) | TOON (Standard) | ETON |
| :--- | :--- | :--- | :--- |
| 可読性 | ◎ (標準的) | ◎ (YAML ライク) | △ (辞書が必要) |
| 圧縮率 | 0% (基準) | ~40% 削減 | **~66% 削減** |
| **設計思想** | 汎用データ交換 | 人間とLLMの共生 | **LLM 通信プロトコル** |
| **状態管理** | Stateless | Stateless | **Stateful** (辞書再利用) |
| **適正データ量** | 小規模 | 中規模 | **大規模・繰り返し** |

## 2. 定量的効率 (Token Efficiency)

100レコード程度のユーザー情報を送信する場合のトークン消費推計：

- **JSON**: 2500 tokens (キーの重複が激しい)
- **TOON**: 1500 tokens (キーを固定して CSV 表示)
- **ETON**: **800 tokens** (キーと頻出値を辞書化)

## 3. なぜ ETON なのか？

- **コンテキストの節約**: 大規模な RAG システムやマルチエージェント通信において、トークン消費（コスト）は最大のボトルネックです。
- **意味の分離**: 構造（スキーマ）と値（辞書）を分離することで、データ本体の純粋な「差分」のみを高速・低コストに転送できます。

## 4. 使い分けのガイドライン

1.  **JSON**: 構造が動的で、可読性と互換性が最優先される場合。
2.  **TOON**: 人間も LLM も読み書きし、かつ基本的な圧縮も必要な場合。
3.  **ETON**: **システムから LLM への一方的な大量データ入力**において、コストと速度を究極まで追求する場合。
